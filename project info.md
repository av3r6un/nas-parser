# NAS Parser

Production ETL-парсер прайс-листов NAS Crystal.

---

# Цель проекта

Проект автоматически обрабатывает Excel-прайсы NAS Crystal и формирует Excel-файл импорта для 1С.

Основные источники:

- 12cut
- 16cut
- K9 (пришивные)

Конечный результат:

```
Excel прайсы
        ↓
Reader
        ↓
Parser
        ↓
ProductRecord
        ↓
Business Enrichment
        ↓
Validation
        ↓
Excel Export
        ↓
1C
```

Проект не является универсальным ETL-фреймворком.

Архитектура строится исключительно под задачи NAS Crystal.

---

# Главные архитектурные принципы

При разработке необходимо соблюдать следующие правила.

## 1. Простота

Не создавать лишних абстракций.

Если задачу можно решить внутри существующего класса — делать именно так.

Не создавать:

- Service
- Factory
- Builder
- Strategy
- Plugin
- Dynamic Discovery

без реальной необходимости.

---

## 2. ProductRecord — центр проекта

Весь pipeline строится вокруг одной модели.

```
ProductRecord
```

Все остальные слои либо создают его, либо обогащают, либо экспортируют.

# Текущая логика

## CutParser

Парсит:

12cut

16cut

Извлекает:

- color
- size
- price
- quantity
- color_code
- cut
- fixation

Создает ProductRecord.

Никакой бизнес-логики внутри Parser нет.

---

## K9Parser

Парсит пришивные стразы.

Поддерживает:

- merged cells
- текущий цвет
- текущую форму

Создает ProductRecord.

fixation всегда

```
sew
```

cut всегда

```
None
```

shape заполняется.

---

# ProductEnricher

Отвечает только за бизнес-обогащение.

Строит:

- sku
- name
- category
- color_code

Не занимается validation.

---

# ProductValidator

Проверяет записи.

Пишет warning/error в RunReport.

Не изменяет ProductRecord.

---

# ExcelExporter

Экспортирует ProductRecord.

Никаких вычислений не делает.

Он только отображает поля модели.

---

# Pipeline

Текущий pipeline

```

↓

↓

или
K9Parser

↓

↓

ProductValidator

↓

```

Pipeline остается координатором.

# Что изменилось в данных

Изначально SKU всегда строился программно.

Теперь структура изменилась.

## Для 12cut / 16cut

В исходных Excel-файлах появился новый столбец:

```
Артикул
```

Он уже содержит готовый итоговый артикул.

Например

```
12cut/Crystal/SS16/Non/001
```

То есть теперь программа НЕ должна вычислять этот артикул повторно.

Он уже существует.

---

Для K9

Никаких изменений нет.

Артикула в Excel нет.

Он по-прежнему должен строиться программно.

---

# Новая концепция SKU

Поле

```
ProductRecord.sku
```

теперь означает

> Итоговый артикул товара.

Не важно, каким способом он получен.

Возможны два варианта.

## Вариант 1

Артикул прочитан из Excel.

Используется для:

- 12cut
- 16cut

---

## Вариант 2

Артикул построен ProductEnricher.

Используется для:

- K9

---

Таким образом Export вообще не знает происхождение артикула.

Он просто экспортирует готовое поле.

---

# Что необходимо изменить дальше

## 1.

CutParser

должен читать новый столбец

```
Артикул
```

и сразу записывать его в

```
ProductRecord.sku
```

Если артикул заполнен.

---

## 2.

ProductEnricher

не должен всегда строить SKU.

Логика должна стать такой:

```
if record.sku is not None:

    ничего не делать

иначе

    построить sku
```

Таким образом:

12cut

↓

готовый артикул

↓

ничего не вычисляется

---

K9

↓

артикула нет

↓

SKU строится автоматически.

---

## 3.

В будущем аналогичный подход можно применить и к Name.

Если однажды имя товара появится в исходном Excel,

Enricher также должен использовать готовое значение вместо генерации.

---

# Что нельзя менять

Нельзя:

- менять архитектуру проекта;
- переносить бизнес-логику в Exporter;
- переносить validation в Pipeline;
- переносить Excel-логику в ParserBase;
- объединять CutParser и K9Parser;
- создавать новые слои без необходимости;
- усложнять проект ради "красивой архитектуры".

---

# Стиль разработки

Перед каждым изменением необходимо соблюдать следующие принципы.

- минимальный diff;
- не ломать существующие тесты;
- не менять публичные контракты без необходимости;
- сохранять совместимость;
- не выполнять рефакторинг ради рефакторинга;
- сначала предложить изменение, затем реализовывать.
