# Спецификация парсеров

## Общие принципы

- Reader читает Excel как есть и превращает строки в `SourceRow`.
- Parser работает только со структурой конкретного файла.
- Специфика входных колонок не должна протекать в `Pipeline`,
  `ProductEnricher` или `ExcelExporter`.

## 12cut

Поддерживаются файлы, имя которых содержит `12cut`.

Parser извлекает:

- `color`
- `size`
- `price`
- `quantity`
- `color_code`
- `cut="12cut"`
- `fixation` по имени файла
- `sku`, если присутствует столбец `Артикул`

### Обычные размеры

Для размеров вида `SS3`, `SS4`, `SS16`, `SS30` и аналогичных:

- колонка 8: `Артикул`
- колонка 9: `Код цвета`

### Размер `mix`

Для строк, где `size == "mix"`:

- колонка 8: `Код цвета`
- колонка 9: `Артикул`

Это правило знает только `CutParser`.

## 16cut

Поддерживаются файлы, имя которых содержит `16cut`.

Логика идентична `12cut`:

- обычные размеры: `Артикул` в колонке 8, `Код цвета` в колонке 9;
- `mix`: `Код цвета` в колонке 8, `Артикул` в колонке 9.

## K9

Поддерживаются файлы, `stem` которых начинается с `k9`.

Примеры:

- `K9.xlsx`
- `K9_premium.xlsx`

Parser извлекает:

- `color`
- `shape`
- `size`
- `price`
- `quantity`
- `fixation="sew"`
- `cut=None`

### Особенности K9-формата

- допускаются merged cells для цвета;
- допускаются merged cells для формы;
- текущий цвет и текущая форма наследуются последующими строками;
- размер используется как индикатор продуктовой строки;
- quantity и price могут быть заданы Excel-формулами.

### Размеры K9

Встречаются два основных формата:

- `10MM`
- `10*14`

При создании `ProductRecord`:

- `10MM` сохраняется без изменений;
- `10*14` нормализуется в `10x14`.

## Что parser не делает

- не выполняет бизнес-валидацию;
- не экспортирует данные;
- не пишет итоговый Excel;
- не переносит свою Excel-специфику в другие слои;
- не строит лишние абстракции поверх существующей логики.
