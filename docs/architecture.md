# Архитектура

## Общая схема ETL

```text
Excel
  ↓
ExcelReader
  ↓
SourceRow
  ↓
ParserRegistry
  ↓
CutParser / K9Parser
  ↓
ProductRecord
  ↓
ProductEnricher
  ↓
ProductValidator
  ↓
ExcelExporter
  ↓
output/catalog.xlsx
  ↓
CatalogAudit
```

## Общая идея

NAS Parser не является универсальным ETL-фреймворком. Архитектура проекта
собрана под конкретный формат прайс-листов NAS Crystal и сознательно держится
простой.

Центральная модель проекта: `ProductRecord`.

Почти все решения в кодовой базе можно проверить вопросом:

> Этот слой создает, дополняет, проверяет или только отображает `ProductRecord`?

## Ответственность слоев

### Reader

- читает Excel-файлы из `input/`;
- превращает строки листов в `SourceRow`;
- не содержит бизнес-логики каталога.

### Parser

- понимает структуру конкретного входного Excel;
- извлекает поля из строк;
- создает `ProductRecord`;
- не занимается валидацией;
- не переносит Excel-специфику в общую бизнес-логику.

### ProductRecord

- единая модель записи в ETL;
- хранит исходные и вычисленные поля;
- используется всеми следующими слоями.

### Enricher

- достраивает бизнес-поля;
- строит `name`, `category`, `color_code`;
- строит `sku` только если он не пришел готовым из parser.

### Validator

- проверяет целостность и полноту записей;
- пишет предупреждения и ошибки в `RunReport`;
- не изменяет `ProductRecord`.

### Exporter

- отображает `ProductRecord` в формат `catalog.xlsx`;
- не вычисляет бизнес-поля;
- не знает, как именно был получен `sku`.

### Audit

- работает только с готовым `output/catalog.xlsx`;
- не участвует в ETL pipeline;
- не исправляет данные;
- собирает статистику и указывает проблемные строки каталога.

## Parser composition

### CutParser

Поддерживает:

- `12cut`
- `16cut`

Отвечает за:

- `color`
- `size`
- `price`
- `quantity`
- `color_code`
- `cut`
- `fixation`
- готовый `sku`, если во входном файле присутствует столбец `Артикул`

### K9Parser

Поддерживает файлы, `stem` которых начинается с `k9`, например:

- `K9.xlsx`
- `K9_premium.xlsx`

Отвечает за:

- `color`
- `shape`
- `size`
- `price`
- `quantity`
- `fixation="sew"`
- `cut=None`

## Инварианты архитектуры

- `ProductRecord` остается центром проекта.
- `Pipeline` остается координатором.
- `Audit` существует отдельно от ETL.
- Все изменения по формату входных Excel локализуются в parser-слоях.
- Все изменения по бизнес-представлению локализуются в enrichment-слое.

## Что запрещено

- Parser не переносит бизнес-логику в общий базовый класс без необходимости.
- Parser не должен вычислять лишние бизнес-поля, если они уже пришли из Excel.
- Exporter не содержит бизнес-логики.
- Validator не изменяет `ProductRecord`.
- Audit не участвует в ETL.
- Pipeline не превращается в место для бизнес-правил.
- Новые абстракции не добавляются без явной практической пользы.
