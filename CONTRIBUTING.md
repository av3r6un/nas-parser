# Contributing

## Назначение

Этот документ фиксирует стиль развития NAS Parser после стабилизации
архитектуры.

Проект не развивается как open source framework. Это production ETL под
конкретный формат NAS Crystal, поэтому главная цель изменений:

- сохранить предсказуемость;
- не потерять накопленные решения;
- не ломать рабочую архитектуру.

## Основные правила

- Делать минимальный diff.
- Не менять архитектуру без отдельного обоснования и согласования.
- Не ломать существующие публичные контракты без необходимости.
- Один слой отвечает только за свою зону ответственности.
- Сначала локальное решение, затем рефакторинг только если он действительно
  нужен.
- Не добавлять новые абстракции без практической пользы.
- Сначала предложить изменение, затем реализовывать его.

## Ответственность слоев

- Reader читает Excel и не знает бизнес-правил.
- Parser знает только структуру конкретного файла.
- Enricher строит бизнес-поля.
- Validator проверяет, но не исправляет.
- Exporter только отображает готовые данные.
- Audit диагностирует результат, но не участвует в ETL.

## Что особенно важно не делать

- не переносить бизнес-логику в `ExcelExporter`;
- не переносить validation в `Pipeline`;
- не делать `ParserBase` местом для Excel-специфики конкретного формата;
- не объединять `CutParser` и `K9Parser` ради симметрии;
- не добавлять service/factory/strategy/plugin слои без реальной выгоды.

## Как вносить изменения

1. Сначала понять, к какому слою относится задача.
2. Проверить, можно ли решить ее локально.
3. Добавить или обновить тест рядом с изменяемым поведением.
4. Убедиться, что существующие тесты проходят.
5. Не расширять change scope без необходимости.
